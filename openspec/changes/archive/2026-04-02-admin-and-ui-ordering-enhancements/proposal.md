# Proposal: 管理员界面与导入/排序增强

## Overview

本变更覆盖 8 项需求：

1. 设置页新增“导入项目”“导入数据库”，并将导入/导出按钮按两列纵向分组。
2. 隐藏选项、单位、字段、表单、访视页面中的 code 显示，仅隐藏展示，不改数据行为。
3. 为选项、单位、字段、表单、访视补齐拖拽排序，拖拽后自动调整序号。
4. 项目列表中删除按钮常显，并新增复制按钮。
5. 修复项目列表左侧折叠按钮无效。
6. 表单界面相关文案统一为“Code（域名）”。
7. 新增管理员界面，管理员用户名来自 config。
8. 管理员界面支持用户列表、新增用户、删除用户、修改用户名。

本 proposal 只输出 research 阶段得到的约束、边界、风险和验收判据，不进入实现。

---

## Enhanced Requirements

### 需求 1 — 设置页新增导入能力，并重排导入/导出布局

**目标**
- 在设置页新增“导入项目”“导入数据库”两个入口。
- 将数据管理区布局调整为两列纵向分组：左列两个导入，右列两个导出。

**已确认语义**
- “导入项目”输入来源：**单项目 `.db` 文件**。
- “导入数据库”语义：**将外部整库合并进当前库**，不是整库替换。

**现状边界**
- 前端设置弹窗当前只有“导出整个数据库”“导出当前项目”。
- 后端当前没有“导入整库”或“导入单项目数据库”的现成 API。
- 现有 `import_template` 只支持从模板库 `.db` 中选择表单导入，不等同于本需求。

**技术约束**
- 导入项目与导入整库不能复用现有模板导入语义；需要独立处理项目树、owner 绑定、冲突策略。
- 单项目数据库导出文件当前会清空 `user` 表并将 `project.owner_id` 置空；导回主库时必须为导入后的项目重新绑定当前用户，否则会产生孤儿项目。
- 整库合并必须定义冲突处理规则（项目重名、code/唯一键、排序作用域、资源复用与去重），否则不可直接落地。
- 现有导入/导出职责已拆分为 `export` / `import_template` / `import_docx`，新能力不应将这些职责重新混成一个不可维护的大接口。

**验收标准**
- [ ] 设置页数据管理区展示四个按钮：左列“导入项目”“导入数据库”，右列“导出当前项目”“导出整个数据库”。
- [ ] “导入项目”可接收单项目 `.db` 文件，并为当前登录用户生成可见项目。
- [ ] “导入数据库”执行合并而非覆盖当前库。
- [ ] 导入后刷新列表即可看到新项目；不得产生 `owner_id` 为空且当前用户不可见的项目。

### 需求 2 — 隐藏前端所有 code 显示，但不改变行为

**目标**
- 在选项、单位、字段、表单、访视相关页面隐藏 code 展示。
- 仅隐藏显示；数据、接口、导出、引用关系均保持原样。

**现状边界**
- code 显示分散在表格列、行内文本、表单标签等多个位置。
- 字段页实际展示的是 `variable_name`，当前文案为 `Code（变量名）`。
- 表单页存在 `Code` 列与创建/编辑表单对话框中的 `Code` label。

**技术约束**
- 这是纯前端展示调整；后端 schema 与 API 继续返回 code/domain 等完整字段。
- 不得删除、重命名、停用后端字段；否则会影响导出、复制、引用检查及既有编辑逻辑。
- 需要逐页枚举可见展示点，避免只隐藏表格列而漏掉行内文本或对话框字段。

**验收标准**
- [ ] 选项、单位、字段、表单、访视页面中不再显示 code 相关列或文本。
- [ ] 创建、编辑、保存、导出、引用检查行为保持不变。
- [ ] API contract 不变，前端仅改变展示层。

### 需求 3 — 为核心列表补齐拖拽排序，并在拖拽后自动重排序号

**目标**
- 为选项、单位、字段、表单、访视补齐拖拽排序。
- 拖拽后自动按新的可见顺序重排并持久化。

**现状边界**
- 选项页右侧 option 列表已经支持拖拽。
- 单位页已经完整使用 `vuedraggable + useOrderableList`。
- 字段、表单、访视列表仍主要依赖手工输入序号。
- 访视内表单关联（`visit_form.sequence`）也存在独立排序，但不在本轮需求原文的最小明确范围内。

**技术约束**
- 前端应优先复用 `useOrderableList` 的现有交互模式，而不是新增另一套拖拽协议。
- 后端排序基础设施已存在：`order_index` 与 `sequence` 都有现成重排能力，应复用而非重写。
- 作用域唯一性必须保持：同一项目/父级作用域内的 `order_index` 或 `sequence` 不能重复。
- drag-and-drop 的成功判据不只是界面变化，还要求刷新后顺序稳定、导出与预览顺序同步。

**范围边界**
- 本轮明确对象：字典/选项、单位、字段、表单、访视。
- `visit_form` 的嵌套排序是否一并改为拖拽，留待 planning 阶段最终收口。

**验收标准**
- [ ] 字典/选项、单位、字段、表单、访视的主列表支持拖拽排序。
- [ ] 拖拽后自动更新序号，无需用户再手填序号修正。
- [ ] 刷新页面后顺序保持一致。
- [ ] 相关导出/预览顺序与拖拽结果一致。

### 需求 4 — 项目列表删除按钮常显，并新增复制按钮

**目标**
- 删除按钮从 hover 才显示改为常显。
- 在项目列表中新增“复制”按钮。

**现状边界**
- 项目列表已有删除能力，但按钮当前不是常显。
- 后端当前没有项目复制接口。
- `ProjectRepository.get_with_full_tree()` 已具备读取完整项目树的基础能力。

**技术约束**
- “复制项目”不能只复制 project 主表，否则会破坏 forms / visits / fields / codelists / units / form_fields / visit_forms 等引用关系。
- research 结论倾向于**完整项目树深拷贝**；若 planning 阶段要降级为部分复制，必须显式重新界定哪些关系被裁剪。
- 复制后的新项目必须归属于当前用户，且不应复用源项目主键或跨项目引用。
- logo 是否复制未在本轮确认中再次明确，需在 planning 阶段定稿。

**验收标准**
- [ ] 每个项目项中删除按钮常显。
- [ ] 每个项目项中新增复制按钮。
- [ ] 复制成功后，新项目立即出现在当前用户项目列表中。
- [ ] 复制后的项目可正常打开、编辑、导出，不存在跨项目脏引用。

### 需求 5 — 修复项目列表左侧折叠按钮无效

**目标**
- 让“项目列表”左侧的折叠按钮真正控制侧边栏收起/展开。

**现状边界**
- 当前界面只有 `Fold` 图标展示，没有点击事件、没有折叠状态、没有宽度切换逻辑。
- 因此当前不是“按钮坏了”，而是“功能尚未接线”。

**技术约束**
- 折叠状态需要在 App 壳层维护，并与项目选中、项目操作区展示协同。
- 收起后仍需保证项目切换、新增、复制、删除等操作不会误触或失效。

**验收标准**
- [ ] 点击折叠按钮可在展开/收起之间切换。
- [ ] 收起/展开后，项目列表与主内容区布局正常。
- [ ] 项目新增、复制、删除、选中行为不受影响。

### 需求 6 — 表单界面文案统一改为“Code（域名）”

**目标**
- 将表单界面中相关 `Code` 文案统一改为 `Code（域名）`。

**现状边界**
- 已读代码片段中未发现真正的 `Ccode`；当前看到的是 `Code`。
- 因此 research 将该需求解释为：修正/统一表单界面的可见文案，而不是改数据字段。

**技术约束**
- 仅改前端文案，不改后端 schema、字段名、数据库结构。
- 需覆盖表单列表列头、创建/编辑对话框 label 等所有相关展示点。

**验收标准**
- [ ] 表单界面可见的相关 `Code` 文案统一为 `Code（域名）`。
- [ ] 表单创建、编辑、保存逻辑不变。

### 需求 7 — 增加管理员界面，管理员用户名来自 config

**目标**
- 在 config 中配置管理员用户名。
- 当前登录用户名等于该配置值时，前端展示管理员入口，并可进入管理员界面。

**用户确认**
- 管理员模型选择为：**仅用户名入口**。
- 即：本轮不引入真实管理员鉴权，不增加独立凭证，仅按用户名判断是否显示/允许进入管理员界面。

**现状边界**
- 当前 auth 采用“无密码登录 + 用户不存在自动创建”。
- 当前配置模型中没有 admin 配置段。
- 当前没有 admin router / users router / admin 页面。

**技术约束**
- 该方案只适合受信任场景下的便捷入口，不构成真正安全边界。
- 由于任意人都可用任意用户名调用当前登录接口，因此任何知道 `admin_username` 的人都可能冒名进入管理员界面。
- 若未来需要真正的管理员权限控制，必须重新设计 auth/config/dependency，而不是沿用本方案。

**验收标准**
- [ ] config 中可配置 `admin_username`。
- [ ] 当前登录用户名等于该值时，可见管理员入口并进入管理员界面。
- [ ] 非该用户名时不显示管理员入口。
- [ ] proposal 与实现文档中明确标注：这是 UI/便捷入口，不是强安全模型。

---

### 需求 8 — 管理员界面支持用户管理

**目标**
- 管理员界面支持用户列表、新增用户、删除用户、修改用户名。

**用户确认**
- 删除用户策略：**只要该用户仍拥有项目，就禁止删除**。

**现状边界**
- 当前 `User` 模型只有 `username`，没有 role/is_admin/is_active。
- `Project.owner_id` 直接指向 `User.id`。
- 当前 token 的 `sub` 使用 `username`，用户名修改将影响当前 token 语义。

**技术约束**
- 新增用户必须满足用户名唯一约束。
- 删除用户不得造成项目归属丢失或外键悬空，因此当用户仍有项目时必须阻止删除。
- 修改用户名将影响现有 token 查找语义；旧 token 何时失效需要在 planning 阶段明确，但实现不能让状态不可预测。
- 在当前“仅用户名入口”的模型下，用户管理能力天然不具备强安全保证，需要在 proposal/设计中明确风险。

**验收标准**
- [ ] 管理员界面可查看用户列表。
- [ ] 可新增用户，且用户名唯一。
- [ ] 可修改用户名，且冲突时给出明确错误。
- [ ] 若用户仍拥有项目，则删除操作被拒绝并给出明确提示。

## Discovered Constraints

### Hard Constraints
- `App.vue` 是全局壳层，设置弹窗、顶部导入导出按钮、项目列表与侧边栏状态都在这里汇总；需求 1、4、5 的前端落点集中。
- `useOrderableList` 已定义前端拖拽持久化协议：按当前顺序提交 ID 数组；新增拖拽页面应优先复用。
- 后端 `OrderService` 已覆盖 `order_index` / `sequence` 的插入、移动、批量重排；拖拽排序不是从零设计问题。
- 后端当前没有“导入整库”“导入单项目数据库”“项目复制”“用户管理”“管理员入口”对应接口。
- `config.py` 当前没有 admin 配置结构；如需 `admin_username`，必须扩展配置模型与配置文件。
- 当前登录流 `/api/auth/enter` 对任意用户名都可无密码登录并自动建号；因此“仅用户名入口”的管理员方案天然不安全。
- `Project.owner_id` 与 `User` 强绑定；删除用户必须避免孤儿项目或外键冲突。
- 隐藏 code 只允许发生在前端展示层；后端字段和接口契约必须保持稳定。

### Soft Constraints
- 前端已使用 Vue 3 + Element Plus + `vuedraggable`，同类交互应沿用现有体系。
- `refreshKey` + `inject/watch` 是当前主要数据同步模式，新增管理页与列表更新应兼容该模式。
- `useApi.js` 已有缓存与请求去重逻辑，新增数据管理与用户管理能力需要考虑缓存失效边界。
- 错误提示应沿用现有中文错误模型，不引入另一套风格。

### Dependencies
- `frontend/src/App.vue`：设置页、项目列表、侧边栏折叠、管理员入口的主前端依赖。
- `frontend/src/components/UnitsTab.vue`：其拖拽模式是字段/表单/访视列表改造的前端模板。
- `frontend/src/composables/useOrderableList.js`：拖拽排序的统一前端依赖。
- `backend/src/services/order_service.py`：拖拽排序持久化的统一后端依赖。
- `backend/src/config.py` + 根 `config.yaml`：`admin_username` 的真实配置入口。
- `backend/src/models/user.py` + `backend/src/models/project.py`：用户管理与项目归属的核心模型依赖。
- `backend/src/repositories/project_repository.py`：项目复制的最佳现有读取依赖。
- `backend/src/routers/settings.py`、`backend/src/routers/projects.py`、导入导出相关 service/router：需求 1、4、7、8 的后端落点。

### Risks
- **伪管理员风险**：只要知道 `admin_username`，任何人都可能通过当前无密码登录流程冒充管理员。
- **导库合并风险**：整库合并会碰到项目重名、owner 重建、唯一键冲突、排序作用域冲突、资源去重规则不一致等问题；若无明确合并策略，极易产生脏数据。
- **项目复制风险**：若复制范围不完整，最容易漏掉 `visit_form`、`form_field`、`codelist_option` 或外键映射，导致项目可见但不可用。
- **用户名修改风险**：token 的 `sub=username`，改名后现有 token 如何处理必须在 planning 阶段收口，否则行为会不确定。
- **设置页权限风险**：若把高风险导入能力继续挂在当前全局 settings 能力下，而不额外约束入口，普通用户也可能触发高影响操作。

## Open Questions Carried into Planning

以下问题在本轮 research 中已被收束，但仍需在 planning 阶段定稿具体规则：

1. 整库合并的冲突策略：项目重名时是重命名、跳过还是覆盖？
2. 整库合并时，字典/单位/字段等公共资源是按自然键复用、复制，还是由用户手动选择？
3. 项目复制的最终复制范围是否包含 logo，以及命名规则（如是否追加“(副本)”）。
4. 用户名修改后，旧 token 是立即失效还是在下次请求时失效并强制重新登录？
5. 管理员界面仅按用户名入口时，是否需要在 UI 上显式警示“该模式不提供真实安全保护”。
6. `visit_form` 嵌套排序是否纳入本轮拖拽范围，还是只覆盖访视主列表与表单主列表。

- [ ] 所有文档与界面文案必须清楚区分：本轮管理员方案是便捷入口，不是强安全权限体系。
