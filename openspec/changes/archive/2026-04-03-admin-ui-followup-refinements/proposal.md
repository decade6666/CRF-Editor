# Proposal: admin-ui-followup-refinements

## Overview

本变更是对已归档变更 `2026-04-02-admin-and-ui-ordering-enhancements` 的后续收敛，覆盖 9 项新增/修正诉求：

1. 设置中的“导入项目/导入数据库”开放给所有用户。
2. 拖拽排序行为未达到预期，需修正为一致、可持久化的行为。
3. 项目复制按钮常显。
4. 折叠按钮应折叠整个项目选择区，仅保留一个展开按钮。
5. 管理员界面只保留用户管理，不保留其他旧功能。
6. 管理员界面新增“用户项目分批操作”：可将用户项目复制给其他用户，或删除项目；删除后进入回收站并可撤销。
7. 表单右侧预览移除“横向”按钮。
8. 表单右侧预览支持双击字段，弹窗快速编辑字段属性。
9. 模板导入弹窗在选择表单后，支持变量级选择再导入。

本阶段仅输出约束与验收判据，不进入实现。

---

## Enhanced Requirements

### R1 设置导入按钮对所有用户开放

**目标**
- 设置弹窗中的“导入项目(.db)”与“导入数据库(.db)”不再仅管理员可见/可点，普通登录用户也可使用。

**已知现状**
- 前端按钮有 `isAdmin` 控制。
- 后端对应接口在 `backend/src/routers/admin.py`，`/api/admin/import/project-db`、`/api/admin/import/database-merge` 都依赖 `require_admin`。

**硬约束**
- 若“开放给所有用户”按字面执行，则后端门禁必须从 `require_admin` 调整为登录态门禁，且导入结果必须绑定当前用户，不能产生孤儿项目。
- 外部 `.db` 仍需维持 SQLite 头校验、schema 预检、200MB 限制、事务原子性。

**验收标准**
- [ ] 非管理员用户在设置页可见并可触发两类导入。
- [ ] 两类导入成功后，新增项目只归属于当前操作者。
- [ ] 导入失败时不产生脏数据（保持原子性）。

---

### R2 拖拽排序修正为“可预期 + 持久化一致”

**目标**
- 拖拽后顺序与序号、刷新结果、导出顺序一致，不出现回弹/错位。

**已知现状**
- 项目级排序大量走 `OrderService`（完整 ID 列表重排协议）。
- 表单内字段排序由 `FormFieldRepository.reorder` 独立实现，语义与 `OrderService` 有差异。
- 前端存在拖拽与手改序号并存场景。

**硬约束**
- 前后端必须统一排序协议（作用域、完整列表、序号基准、错误处理）。
- 过滤/搜索态拖拽禁用规则需一致，避免“部分列表重排”导致错序。

**验收标准**
- [ ] 拖拽后刷新仍保持相同顺序。
- [ ] 拖拽结果与导出/预览顺序一致。
- [ ] 手工改序号与拖拽不会互相覆盖造成回弹。

---

### R3 复制按钮常显

**目标**
- 项目列表中的复制按钮持续可见，不依赖 hover。

**硬约束**
- 仅调整显示逻辑，不改变项目复制后端语义（深拷贝 + owner 重绑 + 名称冲突规则）。

**验收标准**
- [ ] 每个项目项复制按钮常显。
- [ ] 复制行为与现有后端接口返回一致。

---

### R4 折叠按钮折叠整个项目选择区

**目标**
- 点击折叠后，左侧项目选择区整体收起，仅留一个展开入口。

**硬约束**
- 折叠不应破坏项目切换、复制、删除、刷新、导入等行为。
- 收起态需避免残留交互热区导致误触。

**验收标准**
- [ ] 收起后仅显示展开按钮。
- [ ] 展开后恢复原项目列表与交互能力。

---

### R5 管理员界面仅保留用户管理

**目标**
- 管理员页聚焦用户管理，不承载历史杂项功能。

**硬约束**
- 用户管理最小闭环保留：列表、创建、改名、删除限制（有项目不可删）。
- 与登录态判定接口 `GET /api/auth/me` 保持兼容。

**验收标准**
- [ ] 管理员页仅展示用户管理主流程。
- [ ] 旧功能入口移除或迁移，不留死链。

---

### R6 管理员新增用户项目分批操作 + 回收站可撤销

**目标**
- 管理员可按批次处理用户项目：
  - 复制项目到其他用户；
  - 删除项目；
  - 删除进入回收站并支持撤销恢复。

**已知现状**
- 当前 `Project` 为硬删除级联，无 `deleted_at`、无回收站模型、无恢复接口。

**硬约束**
- 回收站必须引入可追踪删除状态（项目及关联资源一致性）。
- 批处理需事务边界明确：部分失败时要么回滚，要么返回逐条结果并可重试。
- 恢复时需处理名称冲突、owner 冲突与引用完整性。

**验收标准**
- [ ] 支持按条件批量选择项目并执行复制/删除。
- [ ] 删除后项目从主列表消失，进入回收站可见。
- [ ] 回收站可撤销恢复，恢复后项目完整可用。

---

### R7 表单预览移除“横向”按钮

**目标**
- 在表单右侧预览区域去除“横向”按钮。

**硬约束**
- 仅 UI 行为收敛；不改变字段布局数据结构与导出渲染规则。

**验收标准**
- [ ] 预览区域不再出现“横向”按钮。
- [ ] 预览渲染正常，无布局异常。

---

### R8 预览双击字段快速编辑

**目标**
- 双击右侧预览字段，弹出快速编辑弹窗，支持关键字段属性修改。

**硬约束**
- 快速编辑必须与现有字段编辑来源一致，避免同字段两套真值。
- 弹窗提交后需与主编辑区状态同步（含校验与错误提示）。

**验收标准**
- [ ] 双击字段可打开快速编辑弹窗。
- [ ] 保存后预览与主表单数据立即一致。

---

### R9 模板导入支持变量级选择

**目标**
- 在模板导入弹窗中：先选表单，再选变量（字段）后执行导入。

**已知现状**
- 现接口 `ImportExecuteRequest` 仅支持 `source_project_id + form_ids`，不支持字段级 payload。

**硬约束**
- 后端导入契约需扩展为变量级输入，同时保证单位、字典、字段定义依赖闭包完整。
- 变量级过滤后，表单字段顺序与 inline/logline 规则仍需可预期。

**验收标准**
- [ ] 用户可在所选表单下勾选变量子集。
- [ ] 导入后仅包含所选变量及必要依赖资源。
- [ ] 不产生孤立引用与重复冲突脏数据。

## Discovered Constraints

### Hard Constraints
- 管理员权限当前基于 `username == config.admin.username` 的弱门禁；不是角色体系。
- 项目归属隔离依赖 `owner_id`，跨用户操作必须显式设计并校验。
- 项目删除当前是硬删除级联；“回收站+恢复”不是小修，需要数据层能力。
- 模板导入执行接口当前不支持字段级选择，必须改请求模型与导入服务。
- 排序实现存在“项目级统一服务 vs 表单字段独立实现”分叉，行为统一前不可保证体验一致。

### Soft Constraints
- 前端应复用现有 `useOrderableList` / `useSortableTable` 交互协议，避免新旧两套拖拽语义。
- 后端应沿用现有 service/repository 分层，不把复杂逻辑塞进 router。
- 导入、复制、恢复相关流程必须继续保持可测试性（集成测试优先）。

### Dependencies
- 权限链路：`backend/src/dependencies.py` (`get_current_user`, `require_admin`, `verify_project_owner`)。
- 导入链路：`backend/src/routers/admin.py`, `backend/src/services/project_import_service.py`。
- 复制链路：`backend/src/services/project_clone_service.py`, `backend/src/routers/projects.py`。
- 模板导入链路：`backend/src/routers/import_template.py`, `backend/src/services/import_service.py`。
- 排序链路：`backend/src/services/order_service.py`, `backend/src/repositories/form_field_repository.py`。
- 前端壳层与管理页：`frontend/src/App.vue`, `frontend/src/components/AdminView.vue`, `frontend/src/components/FormDesignerTab.vue`, `frontend/src/components/TemplatePreviewDialog.vue`。

### Risks
- 将高影响 `.db` 导入开放给所有用户，会放大弱认证模型下的误操作面与数据污染面。
- 回收站改造涉及删除语义迁移，若处理不完整会造成查询漏过滤、恢复失败或残留脏引用。
- 变量级模板导入若依赖闭包计算不完整，会出现 orphan 单位/字典/字段定义。
- 排序协议不统一将继续导致“界面已变、刷新回弹、导出不一致”。

---

## Success Criteria

- [ ] 9 项需求均有可验证行为，且每项都有明确边界（UI-only / 需后端契约）。
- [ ] 导入、复制、删除/恢复、排序四类高影响流程有端到端验证。
- [ ] 管理员页面职责收敛，普通页面不承载管理员批处理复杂度。
- [ ] 变更后不破坏现有项目隔离与导出能力。
