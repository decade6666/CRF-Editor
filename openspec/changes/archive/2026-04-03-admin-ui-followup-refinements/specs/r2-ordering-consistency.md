# Spec: R2 — 拖拽排序收口为一致且可持久化的行为

## Scope
前端：拖拽、手改序号、预览排序显示。
后端：各 reorder 接口与持久化顺序字段。

## Functional Requirements

### FR-2.1 顺序真值
- `order_index` 是项目级有序资源的唯一持久化顺序真值。
- 活跃代码路径中不再以 `sort_order` 作为权威排序依据。
- `sequence` 仅保留给关系型业务顺位（如 `visit_form`），不替代 `order_index`。

### FR-2.2 reorder 协议
- reorder 请求必须提交完整作用域 ID 列表。
- 列表缺失、重复、跨作用域或越权时返回 400。
- 成功后同一作用域内顺序必须为 `1..n` 稠密排列。

### FR-2.3 前端交互
- 拖拽与手改序号必须最终汇入同一持久化链路。
- 过滤/搜索态禁用拖拽。
- 拖拽后刷新页面，顺序不得回弹。

### FR-2.4 一致性读取
- 列表、预览、导出、复制、导入、Word 导入后的顺序读取同一顺序真值。

## Acceptance Criteria
- [ ] 拖拽后刷新顺序保持一致
- [ ] 手工改序号与拖拽不会互相覆盖导致回弹
- [ ] 预览与导出顺序和列表顺序一致
- [ ] 任何活跃前端排序逻辑不再读取 `sort_order`
- [ ] 脏 reorder 列表被后端拒绝并返回 400
