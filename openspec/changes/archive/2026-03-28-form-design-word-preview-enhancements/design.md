## Context

当前 change 原本聚焦于表单设计弹窗内的前端 Word 预览增强，但在本轮 review 中已经确认：仅修前端会把关键语义分叉保留下来，尤其是 `trailing_underscore`、`多选（纵向）`、字典复用策略与导入/导出契约。基于本轮规划确认，本 change 现扩展为 **前后端一致性修复**，覆盖以下链路：

- 设计器内 HTML 模拟预览
- 模板库预览与模板导入执行
- Word `.docx` 导入
- Word 导出
- 设计器字段保存校验与本地联动刷新

本次规划已消除以下关键歧义：
- **同名字典复用策略**：仅当选项语义签名完全一致时才允许复用；否则新建导入字典。
- **字典语义签名**：必须同时比较 `order_index/顺序`、`code`、`decode`、`trailing_underscore`。
- **冲突字典命名**：采用 `名称（导入）`、`名称（导入2）` 的递增后缀格式。
- **`多选（纵向）` 的权威来源**：以 `field_type` 为准，布局只是表现，不是持久化真相来源。
- **DOCX 尾部下划线解释规则**：像 `男_` 这样的选项文本默认按**字面文本**处理，不自动转成 `trailing_underscore=1`。
- **本次范围**：明确纳入前端、模板导入、DOCX 导入、导出一致性修复，而不是仅停留在前端预览。

多模型分析结果：
- **Codex（后端/集成）**：给出了可执行的后端一致性约束、风险点与 PBT 属性，尤其指出了同名字典复用和 `多选（纵向）` 的假支持问题。
- **Gemini（前端/集成）**：按要求多次重试，但持续受到容量限制与 CLI `AttachConsole failed` 异常影响，未能返回可用结果。本次前端与集成约束因此以本地代码证据、既有 spec、review 结论和用户确认为准。

## Goals / Non-Goals

**Goals:**
- 在表单设计弹窗内提供实时 Word 风格预览，并与主预览复用同一前端渲染语义。
- 修复选项类字段 `trailing_underscore` 在前端预览中的对齐问题，且不改变选项文本与顺序。
- 把 choice 字段的配置约束收敛为统一规则：四种 choice 类型（`单选`、`多选`、`单选（纵向）`、`多选（纵向）`）在需要字典时遵循同一校验。
- 明确默认值覆盖边界：非表格场景仅对允许的文本/数值类字段开放；不得误用于选项类、日志行、标签等不支持类型。
- 修复快速编辑字典名称后的设计器联动刷新，以及字段单位清空后回读一致性。
- 统一模板导入在 `CodeListOption.trailing_underscore` 上的保真行为。
- 统一模板导入、DOCX 导入与导出对 `多选（纵向）` 的字段类型语义，避免“字段类型存在但导出/导入退化”。
- 为同名字典复用、导入执行、预览、导出补齐自动化验证与 PBT 不变式。

**Non-Goals:**
- 不整体重写导出系统、导入系统或表单设计器架构。
- 不新增新的后端实时预览 API。
- 不引入新的全局状态管理、事件总线或第二套预览数据结构。
- 不把 DOCX 中末尾 `_` 文本默认解释为结构化尾线语义。
- 不实现多值默认值、模板表达式、规则引擎或新的字段模型。
- 不恢复与当前需求无关的独立 CRUD / Tab 能力。

## Decisions

### 1. 当前 change 升级为前后端一致性修复，而非继续保持 frontend-only
- **Decision**: 本次规划不再把 `backend/src/services/export_service.py`、模板导入与 DOCX 导入排除在外；凡是会改变预览/导入/导出语义一致性的链路，都纳入本次 change。
- **Rationale**: 本轮 review 已证明，若继续坚持 frontend-only，将把同名字典复用、`多选（纵向）` 退化与导入保真缺口留在系统中，无法达成本次“按建议执行”的目标。

### 2. 同名字典仅在语义签名完全一致时复用
- **Decision**: `ImportService` 在处理同名字典时，必须比较完整 option semantic signature：顺序、`code`、`decode`、`trailing_underscore`。只有完全一致时才复用；否则必须新建导入字典。
- **Rationale**: 名字相同不代表导出语义相同。本次已确认 `trailing_underscore` 属于真实业务语义的一部分，不能在导入时被静默吞掉。
- **Alternatives considered**:
  - 同名一律复用：实现最少，但会静默改写导出语义，已被否决。
  - 同名冲突即报错：安全但交互更重；当前优先采用自动新建导入字典。

### 3. 冲突字典统一采用“名称（导入）/ 名称（导入2）”命名策略
- **Decision**: 发现同名冲突且不能复用时，系统创建 `名称（导入）`、`名称（导入2）` 等递增后缀字典名。
- **Rationale**: 当前模型不要求 `name` 在项目内唯一，但 UI 层仍需要可区分的名称，避免留下多个不可辨识的同名字典。

### 4. `多选（纵向）` 以 `field_type` 为权威真相来源
- **Decision**: 持久化、预览、导出、导入回读都以 `field_type="多选（纵向）"` 作为权威语义；布局、换行与符号只是其表现和解析线索，不可反客为主。
- **Rationale**: 这样才能避免“入口判断支持了，但真实导出/回读仍退化成普通多选”的伪支持。

### 5. DOCX 中末尾下划线按字面文本处理，不隐式恢复为 `trailing_underscore`
- **Decision**: 在 DOCX 导入中，如果选项文本末尾出现 `_`，默认视为 decode 的字面文本一部分，而不是结构化 metadata。
- **Rationale**: 用户已明确要求该规则，说明 DOCX 输入无法稳定区分“文本下划线”和“元语义下划线”。若强行自动转换，会引入误判。
- **Implication**: `trailing_underscore` 的严格保真目标适用于模板 DB 预览/导入链路；DOCX 导入只需保证不误转义、不伪造 metadata。

### 6. 模板库预览、模板导入执行与设计器预览应共享可无损映射的 option 语义
- **Decision**: 与 choice 相关的接口和内部处理至少要共享一套可无损映射的结构化语义：`code`、`decode`、`trailing_underscore`、顺序。
- **Rationale**: 不能继续让一个入口返回对象、另一个入口只返回字符串、第三个入口靠文本后缀隐式编码；这会扩大前后端分支和测试盲区。

### 7. 默认值覆盖边界继续受字段业务语义约束
- **Decision**: 非表格默认值入口和预览覆盖能力，继续只对当前确认的支持字段类型开放；选项类、日志行、标签和不支持类型不得因为本次后端扩展而被放开。
- **Rationale**: 本次新增的后端一致性修复不能反向破坏前端已确认的边界。

### 8. 自动化验证以 PBT / 可验证不变式为基线
- **Decision**: 本次规划要求 tests 不再只验证 helper 或单一接口，而要覆盖以下不变式：
  - 同名字典复用不会改变来源导出语义；
  - `多选（纵向）` 在导入/导出/预览中保持类型与布局语义一致；
  - 模板 DB 中的 `trailing_underscore` 在预览和执行导入后保持不变；
  - DOCX 中的字面 `_` 不被错误转换成 metadata；
  - `unit_id: null` 回读后仍为空；
  - 设计器重复打开/保存不会引入额外 UI 漂移。
- **Rationale**: 这是让 implementation 真正机械执行的最小验收基线。

## Risks / Trade-offs

- **[范围扩大导致实现面上升] → Mitigation**: 仍然限定在预览 / 导入 / 导出的契约收敛，不重构系统架构。
- **[同名字典冲突后自动新建可能增加字典数量] → Mitigation**: 通过稳定后缀命名让用户可识别；优先保证语义正确，后续再治理字典整洁度。
- **[DOCX 字面下划线规则与模板 DB 语义不同] → Mitigation**: 明确记录两条链路的不同不变式，避免测试和实现混用。
- **[`多选（纵向）` 修补只停留在局部入口] → Mitigation**: tasks 必须覆盖前端校验、后端导出、DOCX 导入与回归测试四个层面。
- **[Gemini 未返回导致前端维护性盲点] → Mitigation**: 把前端改动限定为最小增量，且所有新增约束都以现有组件/渲染逻辑复用为前提。

## Migration Plan

1. 先更新 design/specs/tasks，把本次 change 的范围从 frontend-only 改为前后端一致性修复。
2. 实施时先处理 choice 语义边界与校验收敛：`多选（纵向）`、字典必填、同名字典复用签名。
3. 随后补齐模板导入、DOCX 导入、导出与前端预览的结构化 option 语义和测试覆盖。
4. 最后修复设计器联动刷新、单位清空、默认值边界，并补上验证闭环。
5. 本次无数据库迁移脚本；若实施失败，回滚策略为撤销本次局部契约改动，不影响基础 CRUD 能力。
