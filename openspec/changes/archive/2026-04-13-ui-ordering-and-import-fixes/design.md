# UI Ordering and Import Fixes — Design

## Context

当前变更围绕 4 个已确认缺陷收敛为一次纯修复型 planning：

1. 项目列表侧边栏中的复制按钮对比度不足。
2. “字段”与“表单”界面在拖拽或手动改序号后，序号显示与后端真值同步不一致。
3. 模板导入预览弹窗需要升级为真正的双栏结构，并且左侧预览要与 `FormDesignerTab.vue` 右侧预览保持强一致。
4. 项目 `.db` 导出后再次导入时，同名项目应允许导入并按 `原名_导入`、`原名_导入2` 递增命名，同时要修复 `FormField NULL identity key` 的根因，但兼容承诺只覆盖**修复后新导出的项目 `.db`**；历史坏导出只要求稳定拒绝。

新增用户决策进一步收紧了边界：

- 模板预览 / 导入运行时**严格只读**，不能再修改用户提供的模板 `.db`。
- 旧模板库兼容通过**单独迁移脚本**解决，脚本只覆盖模板库 `.db`，并且必须**输出新文件**而不是原地覆盖。
- 项目 `.db` 导入错误契约固定为稳定 JSON，至少包含 `detail` 和 `code`。

## Multi-Model Analysis Summary

- **Codex（后端）**
  - 认可排序问题继续保持后端契约不变，前端统一交互即可。
  - 认定当前最大风险不在命名或错误提示，而在 `backend/src/database.py` 的 `form_field` 迁移路径：现有 `CREATE TABLE AS SELECT` 重建方式会破坏主键、外键、非空和 `order_index` 语义，足以解释 `FormField NULL identity key` 根因。
  - 认定模板预览当前通过补列修改外部 `.db` 的做法不可接受，运行时应改为严格只读。
- **Gemini（前端 / 集成）**
  - 认可模板预览应直接复用 `FormDesignerTab.vue` 依赖的展示语义，而不是继续依赖 `SimulatedCRFForm` 的扁平模型。
  - 强调前端仍需要稳定 JSON 错误契约，不能继续依赖对纯文本 / HTML 错误体的脆弱兜底。
- **歧义审计结论**
  - `NULL identity key` 根因修复必须正式写入本 change，而不是只停留在命名和 JSON 错误体。
  - 必须写死“历史坏项目导出不自动兼容，但要稳定拒绝”的边界。
  - 必须明确模板运行时严格只读，旧模板兼容只走外部迁移脚本。

## Goals / Non-Goals

### Goals
- 将复制按钮对比度修复限制在侧边栏项目列表上下文，不影响其他 link 按钮。
- 统一“字段”“表单”两个页面的排序行为：过滤态禁止排序；完整列表态下，拖拽与手动改序号都走相同的本地重排 + reorder + reload 流程。
- 将模板导入预览升级为真正的设计器语义预览：左侧复用 `FormDesignerTab.vue` 的分组、结构行、样式与控件语义；右侧显示所有可导入项并允许勾选。
- 将模板预览 / 导入运行时改为严格只读；对旧模板库的兼容迁移外移为单独脚本，输出新的兼容 `.db` 文件。
- 将项目 `.db` 导入命名规则固定为 `原名_导入`、`原名_导入2`、`原名_导入3`……，查重仅限当前用户未删除项目。
- 修复宿主数据库 `form_field` legacy 迁移 / 导出根因，确保**修复后新导出的项目 `.db`** 可再次导入，不再触发 `FormField NULL identity key`。
- 保证项目 `.db` 导入所有应用内失败路径都返回稳定 JSON（`detail` + `code`），并维持失败零残留与失败重试幂等。

### Non-Goals
- 不新增新的排序 API 形态，不把现有 reorder 接口改成返回完整列表。
- 不整体改造复制项目、恢复项目或模板 codelist 冲突命名规则。
- 不自动修复已经历史导出的坏项目 `.db`；这类样本只要求稳定拒绝。
- 不在运行时通过 `ALTER TABLE` 或其他方式修改用户原始模板库 `.db`。
- 不把本次变更扩展为通用迁移框架或全局异常系统重构。

## Decisions

### 1. 复制按钮对比度只在侧边栏上下文收敛
- **Decision**：仅对 `.project-item` / `.project-actions` 中的复制按钮建立默认态、hover 态、active 态的对比度规则。
- **Rationale**：复制与删除共用按钮基类，如果改全局 link 按钮样式，容易误伤 danger 语义与其他区域。

### 2. 排序的唯一持久化入口仍然是现有 reorder 接口
- **Decision**：拖拽与手动改序号都先在前端本地重排并归一化 `order_index=1..n`，再提交完整 ID 序列到既有 reorder 端点，并在成功 / 失败后 reload 对齐。
- **Rationale**：后端 `order_index` 真值和批量重排算法已存在，问题只在前端两条入口行为不一致。

### 3. 过滤态禁止任何排序输入
- **Decision**：字段页与表单页在搜索 / 过滤态下，拖拽与手动序号输入一律禁用。
- **Rationale**：reorder 需要完整、无重复、同作用域的 ID 集合，过滤态提交子集天然违约。

### 4. 模板预览改为“设计器语义强一致”数据契约
- **Decision**：模板预览 DTO 必须覆盖 `FormDesignerTab.vue` 左侧预览所需的结构与样式字段，包括：
  - 源 `form_field.id`、`form_id`、`source_project_id`、`order_index`
  - `is_log_row`、`label_override`、`help_text`、`default_value`、`inline_mark`
  - `bg_color`、`text_color`
  - 嵌套 `field_definition` 预览所需字段与 codelist / unit 语义
- **Rationale**：当前扁平 DTO 无法表达设计器预览真正依赖的结构行与样式语义。

### 5. 模板运行时访问严格只读
- **Decision**：模板预览 / 导入在运行时**绝不修改用户提供的模板 `.db`**。任何兼容补齐、结构归一化或旧库修复都不能直接作用于源文件。
- **Rationale**：当前运行时补列会污染模板资产、破坏只读预期，并在只读目录 / 共享盘 / 锁文件场景下直接失败。

### 6. 旧模板兼容改为外部迁移脚本，且输出新文件
- **Decision**：提供一个只针对**模板库 `.db`** 的迁移脚本；脚本读取旧模板库，生成一个新的兼容 `.db` 文件，原文件保持不变。
- **Rationale**：用户已明确选择“模板库单独转化脚本 + 输出新文件”，这也是最安全、最可回退的方式。
- **Runtime consequence**：模板预览 / 导入遇到不兼容旧模板库时，不尝试在线修复，而是返回稳定 JSON 错误，引导用户先运行迁移脚本。

### 7. `field_ids` 语义固定为源 `form_field.id`
- **Decision**：右侧可勾选项统一抽象为“可导入项”，包括普通字段、标签行、日志行；执行导入时提交的 `field_ids` 全部使用源 `form_field.id`。
- **Rationale**：这与现有导入执行逻辑兼容，也能表达没有 `field_definition` 的结构性行。

### 8. 导入执行结果必须与预览语义一致
- **Decision**：导入执行时必须复制所有影响预览表现的 `FormField` 属性，包括 `is_log_row`、`label_override`、`help_text`、`default_value`、`inline_mark`、`bg_color`、`text_color`，并保持选中子集的相对顺序不变。
- **Rationale**：否则预览即使正确，真实导入结果仍会偏离。

### 9. 项目导入命名规则固定为 `_导入` 递增
- **Decision**：无冲突保留原名；首次冲突使用 `原名_导入`；后续冲突使用 `原名_导入2`、`原名_导入3`……；查重范围仅限当前用户的未删除项目。
- **Rationale**：这是用户已确认的业务规则，且与回收站不占名要求一致。

### 10. `FormField NULL identity key` 根因修复只承诺覆盖“新导出”
- **Decision**：本 change 必须修复宿主数据库中会破坏 `form_field` identity / 主键语义的 legacy 迁移路径，确保**修复后新导出的项目 `.db`** 能被再次导入。
- **Rationale**：用户要求覆盖根因修复，但明确只保证新导出文件，不要求自动兼容已经历史导出的坏项目库。

### 11. `form_field` 迁移必须保留主键 / 外键 / 非空 / `order_index`
- **Decision**：移除 legacy `sort_order` 时，禁止继续使用会丢失约束的 `CREATE TABLE AS SELECT` 方案；迁移后的 `form_field` 至少必须保留：
  - `id` 作为 ORM 可识别的 identity 主键
  - `order_index` 真值列
  - 与 `form` / `field_definition` 的关键外键语义
  - 非空 / 唯一等必要约束
- **Rationale**：这是当前 `NULL identity key` 根因链路的核心。

### 12. 历史坏项目导出稳定拒绝，不在线修复
- **Decision**：对于已经历史导出的坏项目 `.db`，导入链路不承诺自动修复；若无法满足当前导入兼容约束，必须在导入前或导入早期稳定返回结构化错误，而不是在 clone / flush 阶段崩溃。
- **Rationale**：这与“只保证新导出”边界一致，也能避免在导入时加入高风险自动修复逻辑。

### 13. 导入错误契约统一为 `detail + code`
- **Decision**：项目 `.db` 导入与模板预览 / 导入的所有应用内失败路径都返回稳定 JSON，至少包含：
  - `detail`: 面向用户的可展示信息
  - `code`: 机器可判断的稳定错误码
- **Rationale**：前端不能继续依赖对纯文本 / HTML / 非结构化异常字符串的脆弱解析。

### 14. 事务边界必须保证失败零残留
- **Decision**：目标库中的命名解析、clone、flush 以及所有持久化步骤都必须处于统一原子事务中；任何失败都不得留下半成品项目、孤儿子资源或占名残留。模板迁移脚本生成的新文件与运行时导入事务边界分离。
- **Rationale**：用户已明确要求失败零残留与失败重试幂等。

## Risks / Trade-offs

- **[模板预览 DTO 变厚] → Mitigation**：只暴露 `FormDesignerTab.vue` 当前实际消费的视图语义，不把 ORM 模型原样泄露给前端。
- **[旧模板兼容外移后，用户多一步操作] → Mitigation**：运行时通过稳定错误码与清晰文案指向迁移脚本；换来真正只读与可回退。
- **[只保证新导出，历史坏项目库仍不可导入] → Mitigation**：文档中写死兼容窗口，并用结构化错误让失败可判定、可说明。
- **[迁移修复聚焦在 `form_field`] → Mitigation**：测试必须覆盖主键、外键、非空和 `order_index` 四类关键不变量，避免“只看导入通过”而漏掉结构退化。

## PBT / Verifiable Properties

1. **排序 round-trip**：任意合法完整 `ordered_ids` 提交 reorder 后，重新 list / reload 读回的顺序与提交顺序一致，且 `order_index` 恰为连续 `1..n`。
2. **排序非法输入不变式**：缺项、重复、跨作用域或未知 ID 的 reorder 请求返回错误，且不会修改原顺序。
3. **过滤态排序禁用**：在搜索 / 过滤态下，拖拽与手动序号输入都不可触发排序请求。
4. **预览-执行一致**：对任意合法导入项子集，左侧预览中的集合与相对顺序，与导入后目标表单中的集合与相对顺序一致。
5. **模板运行时只读**：任意一次模板预览 / 导入失败或成功，都不会改写源模板 `.db` 文件内容。
6. **模板迁移脚本非破坏性**：对任意旧模板库输入，脚本总是产出一个新文件；原文件内容与路径保持不变。
7. **新导出回环**：对修复后的宿主数据库导出的项目 `.db`，执行“导出 → 单项目导入”后，不会触发 `FormField NULL identity key`。
8. **历史坏项目库稳定拒绝**：对不满足当前 `form_field` identity 约束的旧项目 `.db`，导入返回结构化错误（`detail + code`），而不是非结构化崩溃。
9. **命名单调性**：同一用户下重复导入同名项目时，名称按 `原名_导入`、`原名_导入2`、`原名_导入3`……单调扩展且始终唯一。
10. **失败零残留 / 重试幂等**：导入在预检、clone、flush 任一阶段失败后，数据库状态与导入前一致；重复失败不会累积脏数据或占名。

## Implementation Order

1. 固化 `specs/*.md` 与 `tasks.md`，消除剩余决策点。
2. 先修 `form_field` legacy 迁移 / 新导出回环根因，明确“新导出可再导入”的宿主数据库不变量。
3. 再收敛模板预览 / 导入为严格只读路径，并补模板库迁移脚本方案。
4. 然后统一字段页 / 表单页排序交互约束。
5. 最后处理侧边栏复制按钮对比度、项目导入命名与稳定错误体。
