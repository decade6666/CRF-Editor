# Design: admin-auth-sort-fixes

## 1. 目标与范围

本变更聚焦 9 个相互关联的修复点，但总体目标只有一个：**在不改变 FastAPI + Vue 3 + SQLite 现有架构的前提下，把管理员统计、设置权限、导出边界、退出登录、拖拽排序与导入重名行为统一到现代码真实能力之上**。

本设计遵循三条原则：

- **最小必要修复**：优先复用现有路由、服务与排序基础设施，不为本次 change 发明新框架。
- **前后端口径一致**：同一业务真值必须在 UI 显示、接口校验、持久化规则中保持一致。
- **以当前代码为准做裁决**：proposal 中与现实现冲突的描述，必须先落成明确边界，再进入实现。

---

## 2. 已确认的当前事实

### 2.1 管理员用户统计与删除

当前 `backend/src/services/user_admin_service.py` 的真实逻辑是：

- `list_users()` 的 `project_count` 统计全部 `owner_id == user.id` 的项目；**未过滤** `deleted_at`。
- `delete_user()` 的删除前校验同样统计全部 owned projects；**未过滤** `deleted_at`。

因此当前系统里：

- 用户列表中的项目数 = 活跃项目 + 回收站项目；
- 只要用户仍拥有任意软删除项目，也不能删除该用户。

### 2.2 设置权限边界

当前 `backend/src/routers/settings.py` 中：

- `GET /api/settings`
- `PUT /api/settings`
- `POST /api/settings/ai/test`

全部使用 `require_admin`。

当前 `frontend/src/App.vue` 中，`openSettings()` 一打开就请求 `/api/settings`，所以普通用户会直接看到“加载设置失败: 需要管理员权限”。

### 2.3 导出能力边界

当前 `backend/src/routers/export.py` 真实存在两类导出：

- `GET /api/export/database`：管理员整库导出；
- `GET /api/projects/{project_id}/export/database`：当前用户可导出自己拥有的单项目数据库。

当前**不存在**“导出当前用户全部项目聚合数据库”的专门接口。

### 2.4 退出登录边界

当前认证模型为：

- `POST /api/auth/enter` 无密码用户名登录；
- 前端使用 `localStorage` 保存 `crf_token`；
- 未发现后端 logout / token revoke 接口。

因此当前架构下，“退出登录”的最小实现边界是：**前端清 token + 返回登录入口**。

### 2.5 排序基础设施

当前排序基础设施已经比较完整：

- 表格拖拽：`frontend/src/composables/useSortableTable.js`
- 列表拖拽：`frontend/src/composables/useOrderableList.js`
- 后端统一重排：`backend/src/services/order_service.py`

`OrderService` 的关键约束已经明确：

- 请求必须提交**完整作用域 ID 列表**；
- 重复 ID / 缺失 ID / 跨作用域 ID 一律报错；
- `order_index` 与 `sequence` 都有两阶段安全重排能力；
- 排序落库后必须压实为连续序号。

### 2.6 各排序区域现状

#### 项目列表
- 前端：`App.vue` 已接 `useSortableTable`
- 后端：`/api/projects/reorder` + `ProjectRepository.reorder()` 已存在
- 风险点：若后端作用域仍包含软删除项目，而前端只提交活跃项目，则会触发“ID 列表不完整”

#### 字典左侧列表
- 前端：`frontend/src/components/CodelistsTab.vue` 目前仍是手工改 `order_index`
- 后端：`POST /api/projects/{project_id}/codelists/reorder` 已存在
- 结论：主要缺口在前端未接拖拽，而不是后端协议缺失

#### 字典右侧选项列表
- 前端：已使用 `useOrderableList`
- 后端：`POST /api/projects/{project_id}/codelists/{cl_id}/options/reorder` 已存在
- 结论：已具备完整拖拽链路

#### 访视列表
- 前端：`frontend/src/components/VisitsTab.vue` 已接 `useSortableTable`
- 后端：`POST /api/projects/{project_id}/visits/reorder` 已存在
- 结论：已具备拖拽排序能力

#### 访视内表单列表
- 前端：目前只支持手工改 `sequence`
- 后端：只有 `PUT /api/visits/{visit_id}/forms/{form_id}` 单条改序；**没有**批量 reorder 端点
- 结论：这是前后端双侧缺口，不是单纯 UI 漏接

### 2.7 项目导入重名行为

当前 `backend/src/services/project_import_service.py` 已有 `_resolve_import_name()`：

- 原名冲突时自动改为 `原名 (导入1)`、`原名 (导入2)`、...
- merge 导入还会把 renamed 信息写入结果

因此 proposal 中“导入重名会直接失败”的说法，与当前代码事实冲突。后续实现阶段必须先定位真实失败路径，不能把它简单当成功能缺失。

## 3. 关键裁决

### 3.1 R1 用户项目数与用户删除口径

**裁决：本次 change 采用“活跃项目数”口径。**

也就是说：

- 管理员用户列表中的 `project_count` 只统计 `deleted_at IS NULL`；
- 删除用户前校验也只统计 `deleted_at IS NULL`；
- 若用户只剩回收站项目，则允许删除该用户。

**原因：**

1. 这是 proposal 已明确写入并被用户接受的目标语义；
2. 管理员界面当前展示的“项目数”更接近“当前活跃项目负载”，而不是历史垃圾桶占用；
3. 继续把回收站项目计入统计，会出现“批量删除项目后 UI 仍显示有项目、用户无法删除”的体验冲突；
4. restore owner 风险虽然存在，但它属于“用户删除后回收站项目如何处理”的后续约束问题，不应阻止本次对齐“活跃项目数”语义。

**强约束：**

- 列表显示口径与删除校验口径必须同步调整；
- 不允许出现前端显示 0 但后端仍以软删除项目拒绝删除的分裂行为。

### 3.2 R2 退出登录

**裁决：退出登录只做前端会话退出，不新增服务端注销体系。**

行为定义：

- 在设置弹窗中增加“退出登录”按钮；
- 点击后清除 `localStorage` 中的 token；
- 返回登录入口；
- 保留上次用户名输入值。

**不做：**

- token 黑名单；
- 服务端 revoke；
- 多端会话管理。

### 3.3 R5 设置权限

**裁决：保留管理员级 `/api/settings*` 门禁，同时把设置弹窗拆成“用户轻量设置壳层 + 管理员扩展区”。**

行为定义：

- 所有已登录用户都可以打开设置弹窗；
- 普通用户只看到用户侧内容：例如主题、退出登录等；
- 只有管理员才请求 `/api/settings` 并显示模板路径、AI 配置、全局数据管理等内容。

**关键点：**

- “设置弹窗可打开”与“全局设置可读取”是两个层级；
- 本次 change 不放开 `/api/settings` 给普通用户。

### 3.4 R6 导出数据库

**裁决：不放开管理员整库导出；新增“当前用户全部项目聚合数据库导出”能力。**

行为定义：

- 管理员继续保留 `GET /api/export/database` 整库导出；
- 普通用户点击“导出数据库”时，不再命中管理员整库接口；
- 新增一个面向登录用户的“当前用户聚合导出”接口，只导出该用户拥有的项目集合。

**强约束：**

- 普通用户导出的结果中不得包含其他用户项目；
- 单项目导出能力继续保留；
- 前端按钮文案仍可保持“导出数据库”，但调用语义必须按用户身份区分。

### 3.5 R4 / R8 / R9 排序真值

**裁决：所有已有拖拽区域都必须复用当前“完整作用域列表 + 即时持久化 + 连续序号”的统一协议。**

具体含义：

- 拖拽 drop 后立即提交完整 ID 列表；
- 保存成功后序号压实为 `1..n`；
- 刷新后顺序不可回弹；
- 手工改序号与拖拽排序必须汇入同一后端真值。

### 3.6 R4 项目排序作用域

**裁决：项目排序作用域按“当前活跃项目列表”对齐。**

这意味着：

- 前端项目列表只展示活跃项目时，提交给 reorder 的也只应是活跃项目；
- 后端 reorder 的有效作用域也必须只包含活跃项目；
- 回收站项目不参与当前活跃列表排序序列。

否则用户一旦存在软删除项目，就会持续触发“ID 列表不完整”。

### 3.7 R8 字典左侧列表排序

**裁决：复用既有 codelist reorder 接口，只补前端拖拽接线。**

不新增协议，不新增后端接口。

### 3.8 R8 访视内表单排序

**裁决：补齐 VisitForm 批量 reorder 契约，再接前端拖拽。**

原因：

- 当前只有单条 `PUT` 改 `sequence`；
- 若前端直接拖拽后循环调用单条接口，会形成不稳定、多次写回、序号回弹风险；
- 此区域应像其他排序列表一样拥有完整列表批量提交能力。

### 3.9 R7 导入重名问题

**裁决：本次实现以“定位真实失败路径并修复行为不一致”为目标，不重写已存在的重名命名策略。**

也就是说：

- 保留 `_resolve_import_name()` 的 `(...导入N)` 规则；
- 重点排查导出文件内容、导入前校验链路、前端提示翻译与成功提示最终项目名是否一致。

### 3.10 R3 主界面蓝色调浅

**裁决：优先通过主题变量 / 调色板统一微调，不在组件内分散硬编码颜色。**

**强约束：**

- 同时检查浅色 / 暗色模式；
- 保持按钮、边框、选中态、提示的可读性；
- 这是视觉微调，不做大规模 UI 重构。

## 4. API 与行为设计

### 4.1 设置弹窗行为

#### 普通用户
- 可打开设置弹窗
- 不请求 `/api/settings`
- 可见内容：用户轻量设置、退出登录

#### 管理员
- 可打开设置弹窗
- 请求 `/api/settings`
- 可见内容：用户轻量设置 + 模板路径 + AI 配置 + 全局数据管理 + 管理员导出能力

### 4.2 退出登录行为

前端定义：

1. 保留用户名状态；
2. 清理 token；
3. 重置当前登录态；
4. 返回登录入口。

### 4.3 用户聚合导出接口

建议新增登录态接口，例如：

```http
GET /api/projects/export/database
Auth: login required
```

返回：

- 仅包含当前用户全部项目的 SQLite 数据库文件；
- 文件结构兼容现有导入链路。

实现层关键约束：

- 不能复用管理员整库导出结果直接下发；
- 应沿用现有数据库导出服务的快照/裁剪思路；
- 作用域应以当前登录用户拥有的项目集合为准。

### 4.4 项目排序接口

保留现有：

```http
POST /api/projects/reorder
Body: [project_id1, project_id2, ...]
```

但后端有效作用域必须明确为：

- 当前用户的活跃项目；
- 管理员按 `user_id` 查看他人项目列表时，则是该目标用户的活跃项目。

### 4.5 访视内表单排序接口

建议新增：

```http
POST /api/visits/{visit_id}/forms/reorder
Body: [form_id1, form_id2, ...]
```

语义：

- 对 `VisitForm.sequence` 执行完整列表批量重排；
- 作用域限定为当前 visit 下全部已关联表单；
- 保存后压实为连续 `1..n`。

## 5. 风险与缓解

| 风险 | 等级 | 缓解 |
|---|---|---|
| 只修用户列表显示，不修删除校验，导致前后端口径不一致 | 高 | `list_users()` 与 `delete_user()` 同步改为活跃项目口径 |
| 普通用户设置弹窗若仍在打开时请求 `/api/settings`，报错仍然存在 | 高 | 将“打开弹窗”与“加载管理员设置”解耦，按 `is_admin` 条件请求 |
| 普通用户导出若错误复用整库导出，会突破安全边界 | 高 | 新增专门的用户聚合导出接口，严格按 owner 裁剪 |
| 项目 reorder 若后端作用域仍含回收站项目，拖拽持续失败 | 高 | reorder 作用域与当前活跃列表对齐，只统计 `deleted_at IS NULL` |
| VisitForm 仍用单条改序模拟拖拽，会出现多次写回和顺序回弹 | 高 | 新增批量 reorder 接口，前端只在 drop 后提交一次完整列表 |
| 导入重名问题误判为“没能力”，导致重复造轮子 | 中 | 保留现有 `_resolve_import_name()`，只排查真实失败路径 |
| 主色调调整分散到组件内，后续难维护 | 中 | 统一改主题变量，避免局部硬编码 |

## 6. 测试与不变量

| 主题 | Invariant |
|---|---|
| 用户统计 | 用户列表 `project_count` 与用户删除校验都只统计活跃项目 |
| 退出登录 | 清 token 后回到登录入口，且用户名仍保留 |
| 设置权限 | 普通用户能打开设置但不请求管理员接口 |
| 用户聚合导出 | 普通用户导出结果只包含自己的项目 |
| 项目排序 | 活跃项目 reorder 后顺序持久化，刷新不回弹，序号连续 |
| 字典左侧排序 | 拖拽后立即保存，刷新后顺序一致 |
| 访视内表单排序 | 拖拽后 `VisitForm.sequence` 连续且持久 |
| 导入重名 | 同名项目导入不会因名称冲突失败，最终项目名带 `(导入N)` 后缀 |
| 主题微调 | 浅色/暗色模式下对比度不退化 |

---

## 7. 实施顺序

1. 先收口 **用户统计 / 删除校验 / 项目排序作用域**，解决最核心的口径冲突。
2. 再拆分 **设置壳层与管理员设置读取**，补“退出登录”。
3. 然后新增 **用户聚合导出**，保留管理员整库导出边界。
4. 再补 **字典左侧拖拽** 与 **访视内表单批量 reorder + 前端拖拽**。
5. 最后处理 **导入重名真实失败路径** 与 **主题颜色微调**。

此顺序可以先把前后端语义冲突最大的点落稳，再补 UI 与体验项，避免中途反复改 contract。
