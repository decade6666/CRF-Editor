# Proposal: admin-auth-sort-fixes

## Overview

本变更聚焦 9 个相互关联的修复/收敛点，目标是在**不改变现有整体架构（FastAPI + Vue 3 + SQLite）**的前提下，修复管理员用户统计、设置权限、导入导出权限、拖拽排序一致性、登录退出体验与主界面视觉细节问题。

本阶段仅输出 research 产物：增强后的需求、已确认边界、约束集合、风险与可验证成功判据；**不进入实现**。

---

## Enhanced Requirements

### R1 管理员删除项目后，用户管理中的项目数必须立即反映“活跃项目数”

**目标**
- 管理员在用户管理中批量删除项目后，被删除项目对应用户的“项目数”应立即正确减少。
- 若某用户只剩回收站项目，则该用户应允许被删除。

**已知现状**
- 管理员用户列表中的 `project_count` 与删除用户时的项目数校验，当前都来自后端统计。
- 项目已采用软删除（`deleted_at`）进入回收站。

**技术约束**
- “项目数”语义必须与当前活跃项目列表一致，即只统计 `deleted_at IS NULL` 的项目。
- UI 禁用态与后端删除校验必须基于同一统计口径，不能出现“前端显示 0 但后端仍拒绝”的分裂行为。

**验收标准**
- [ ] 管理员删除项目后，重新加载用户列表时，项目数立即按活跃项目数更新。
- [ ] 仅剩回收站项目的用户可被删除。
- [ ] 用户删除接口与用户列表显示口径一致。

### R2 设置中增加“退出登录”按钮，并返回登录入口

**目标**
- 在设置中增加“退出登录”按钮。
- 点击后退出当前登录态，并回到用户名输入入口。

**用户确认**
- 退出后：**保留上次用户名**。

**已知现状**
- 当前认证为无密码用户名登录 + JWT，本地通过 `localStorage` 保存 token。
- 后端未发现独立 logout / token 撤销接口。

**技术约束**
- 在当前架构下，退出登录优先应理解为**前端清除 token 并切换视图**，而不是新增服务端注销体系。
- 如未来要求“旧 token 立即失效”，那是单独的认证扩展，不属于本次最小修复范围。

**验收标准**
- [ ] 设置弹窗中存在“退出登录”入口。
- [ ] 点击后当前 token 被清除，界面返回登录入口。
- [ ] 登录入口保留上次用户名。

### R3 主界面的蓝色整体再浅一些

**目标**
- 在不重做设计系统的前提下，让主界面的蓝色视觉更浅、更柔和。

**已知现状**
- 前端已使用 CSS 变量与主题切换；主色主要集中在统一变量与部分组件样式上。

**技术约束**
- 应优先调整主题变量/调色板，而非在多个组件中分散写死颜色。
- 需同时关注浅色与暗色模式下的对比度，不得因变浅导致按钮/提示不可读。
- 这是**视觉微调**，不应引发大范围 UI 重构。

**验收标准**
- [ ] 主界面蓝色系整体较当前更浅。
- [ ] 主要按钮、选中态、边框与提示仍保持足够对比度。
- [ ] 暗色模式下无明显可读性退化。

### R4 项目列表拖拽排序必须可用，并在松手后立即持久化

**目标**
- 项目列表支持拖拽排序。
- 鼠标松开后立即按当前顺序写回后端并生效。

**已知现状**
- 前端已存在 `useSortableTable` 与项目 reorder 接口。
- 后端项目排序走 `ProjectRepository.reorder()` 与统一排序服务。

**技术约束**
- 排序提交必须满足后端“完整作用域 ID 列表”的契约。
- 若当前作用域包含回收站项目，而前端只提交活跃项目 ID，会导致排序失败；因此前后端作用域必须一致。
- 排序写回后，活跃项目序号应保持连续，不允许跳号。

**验收标准**
- [ ] 项目列表可拖拽排序。
- [ ] 松手后立即持久化。
- [ ] 刷新后顺序不回弹。
- [ ] 当前用户活跃项目序号连续为 `1..n`。

### R5 非管理员打开设置时，不应再报“需要管理员权限”

**目标**
- 普通登录用户可以打开设置。
- 但普通用户不应因此读取或修改管理员级的全局配置。

**用户确认**
- 采用：**用户轻量设置**。
- 即：非管理员可打开设置，但只看到用户侧内容（如退出登录、主题等）；管理员继续看到模板路径、AI 配置、全局数据管理等内容。

**技术约束**
- 不应简单去掉 `/api/settings` 的管理员限制并把全局配置暴露给所有用户。
- 更合理的边界是：
  - 普通用户：可打开用户级设置壳层；
  - 管理员：可读取/修改全局设置。
- 需要区分“设置弹窗可打开”与“全局配置可读取”两个层级。

**验收标准**
- [ ] 非管理员可打开设置弹窗。
- [ ] 非管理员不会再看到“加载设置失败: 需要管理员权限”。
- [ ] 非管理员不可见管理员级全局配置。
- [ ] 管理员仍可正常读取与保存全局设置。

---

### R6 “导出数据库”权限需要调整为用户侧聚合导出，而非整库备份放开

**目标**
- 修复当前导出数据库时报“需要管理员权限”的体验问题。
- 允许普通用户导出**当前用户拥有的全部项目聚合数据库**。

**用户确认**
- “导出数据库”指：**聚合当前用户所有项目的数据库**。
- 不是把整库备份权限开放给所有用户。

**已知现状**
- 当前后端存在两类数据库导出：
  - 整库导出 `/api/export/database`：仅管理员；
  - 单项目导出 `/api/projects/{project_id}/export/database`：普通用户可导出自己的项目。
- 当前并不存在“导出当前用户全部项目集合”的专门接口。

**技术约束**
- 不能把整库备份接口直接开放给普通用户，否则会突破当前安全边界。
- 需要单独界定“当前用户聚合导出”的语义、数据范围与文件结构。
- 当前项目级导出与整库导出语义已经分离；新增能力应延续这种分层，而不是混用管理员整库导出接口。

**验收标准**
- [ ] 普通用户可触发“导出数据库”且不再收到管理员权限错误。
- [ ] 导出结果只包含当前用户拥有的项目数据。
- [ ] 管理员整库导出能力仍与普通用户聚合导出严格区分。
- [ ] 当前项目导出能力继续保留且可用。

### R7 项目导出后再导入若重名，应自动改名而不是失败

**目标**
- 导入项目数据库时，若目标用户下已存在同名项目，不应失败。
- 应自动改名，并追加导入后缀。

**已知现状**
- 当前代码中的 `project_import_service._resolve_import_name()` 已包含导入重名改名逻辑，格式为 `原名 (导入1)`、`原名 (导入2)`。
- 这说明“需求描述中的失败现象”与当前代码存在冲突，需要在 planning / implementation 阶段进一步定位：
  - 可能是单项目导出文件内容与导入契约不一致；
  - 可能是前端提示路径与真实错误不一致；
  - 也可能是其他重复校验路径先于命名修正触发。

**技术约束**
- research 阶段不得假设“当前完全没有该能力”；应以“已有重名策略，但现场行为与代码不一致”为约束。
- 最终实现需保持导入成功后的项目内容完整、owner 绑定正确、顺序合理。

**验收标准**
- [ ] 导入重名项目时不会因为名称冲突直接失败。
- [ ] 导入后的项目名自动追加导入后缀。
- [ ] 前端成功提示反映最终项目名。
- [ ] 导入后的项目可正常打开与编辑。

---

### R8 字典左侧列表、访视内表单列表应补齐拖拽排序

**目标**
- “选项”界面左侧的字典列表支持拖拽排序。
- “访视”界面右侧的访视内表单列表支持拖拽排序。

**已知现状**
- 字典右侧选项列表已支持拖拽。
- 字典左侧列表当前只有手工改序号，未接上拖拽。
- `VisitForm` 当前只有单条更新 `sequence` 的接口，没有与其他列表一致的批量拖拽 reorder 入口。

**技术约束**
- 字典左侧列表应优先复用现有 codelist reorder 能力，而不是新增另一套协议。
- 访视内表单列表若要拖拽，需要补齐后端批量排序契约，保证 `VisitForm.sequence` 可按完整列表即时重排。
- 拖拽与手工改序号都必须遵循同一排序真值，避免回弹。

**验收标准**
- [ ] 字典左侧列表支持拖拽。
- [ ] 访视内表单列表支持拖拽。
- [ ] 松手后顺序立即持久化。
- [ ] 刷新后顺序保持一致。

---

### R9 所有可拖动排序的地方，都要在松手后立即更新连续序号且不允许跳号

**目标**
- 所有已支持拖拽排序的列表，在松手后都要立即更新并持久化当前顺序。
- 排序字段必须是连续序号，不允许跳号。

**范围边界**
- 该要求覆盖项目、字典、字典选项、字段、表单、访视、访视内表单，以及任何当前已有拖拽协议的同类列表。

**技术约束**
- 需要统一拖拽后的后端写回协议与前端刷新策略。
- 对 `order_index` 与 `sequence` 两种排序字段都必须满足“连续且无空洞”。
- 若存在软删除或隐藏项，序号连续性应以**当前活跃作用域**为准，而不是把回收站项目混入当前排序序列。

**验收标准**
- [ ] 所有拖拽列表在 drop 后立即保存。
- [ ] 对应序号字段连续且无跳号。
- [ ] 刷新、重新进入页面、导出/预览后的顺序保持一致。

## Discovered Constraints

### Hard Constraints
- 后端管理员用户项目数统计与删除校验当前都没有过滤 `deleted_at`，因此会把回收站项目误计入“项目数”。
- 项目列表排序当前受限于后端作用域定义：若 reorder 仍把回收站项目视为同一作用域，而前端只提交活跃项目，就会触发“ID 列表不完整”类错误。
- 设置接口 `/api/settings*` 当前明确是管理员接口；普通用户无法直接读取完整全局配置。
- 整库导出 `/api/export/database` 当前明确是管理员能力；不能直接按“修报错”思路整体放开。
- 当前后端没有专门的“当前用户全部项目聚合数据库导出”接口；若用户确认需要该能力，则属于**新增受限导出能力**，不是简单权限放开。
- 当前没有服务端 logout 接口；退出登录在现有架构下更符合“前端清 token + 切换视图”的实现边界。
- 访视内表单（`VisitForm`）当前缺少与其他列表一致的批量 reorder 端点；若要拖拽排序，需要补齐后端能力。
- 所有排序实现都必须兼容 SQLite 与当前手写迁移模式，优先复用现有 `OrderService`。
- 项目导入重名自动改名逻辑在代码中已存在，因此“重名时报错”不能直接当作功能缺失，必须视为行为与现代码不一致的问题来源。

### Soft Constraints
- 前端已形成两套可复用的排序交互：`useSortableTable`（表格）与 `useOrderableList`（draggable 列表）；新增/修复应优先复用，不再新增第三套协议。
- 主题与颜色应优先通过全局 CSS 变量调整，避免组件内散落硬编码颜色。
- 用户级设置与管理员级全局设置应在同一设置壳层中清晰分区，而不是拆成完全不同入口造成认知负担。
- 错误提示保持现有中文风格，并尽量把“权限不足”“导入重名”“排序失败”区分清楚。

### Dependencies
- 认证与退出登录：`frontend/src/App.vue`、`frontend/src/components/LoginView.vue`、`frontend/src/composables/useApi.js`、`backend/src/routers/auth.py`、`backend/src/dependencies.py`。
- 管理员用户统计与删除：`frontend/src/components/AdminView.vue`、`backend/src/services/user_admin_service.py`、`backend/src/routers/admin.py`。
- 项目排序：`frontend/src/App.vue`、`frontend/src/composables/useSortableTable.js`、`backend/src/repositories/project_repository.py`、`backend/src/services/order_service.py`。
- 设置权限：`frontend/src/App.vue`、`backend/src/routers/settings.py`、`backend/src/dependencies.py`。
- 数据库导出：`frontend/src/App.vue`、`backend/src/routers/export.py`、`backend/src/services/export_service.py`。
- 项目导入重名：`frontend/src/App.vue`、`backend/src/routers/projects.py`、`backend/src/services/project_import_service.py`、`backend/src/services/project_clone_service.py`。
- 字典与选项排序：`frontend/src/components/CodelistsTab.vue`、`frontend/src/composables/useOrderableList.js`、`backend/src/routers/codelists.py`、`backend/src/services/order_service.py`。
- 访视内表单排序：`frontend/src/components/VisitsTab.vue`（待规划阶段进一步精确定位）、`backend/src/models/visit_form.py`、`backend/src/routers/visits.py`、`backend/src/services/order_service.py`。
- 主题颜色：`frontend/src/App.vue`、全局样式/主题变量文件。

### Risks
- 若简单去掉 `/api/settings` 的管理员限制，可能把模板路径、AI URL、模型名等全局配置暴露给所有登录用户。
- 若简单放开 `/api/export/database` 给普通用户，会把整库备份权限错误地下放，而非实现“当前用户项目聚合导出”。
- 若只修前端项目数显示，不修后端删除校验，仍会出现“显示为 0 但无法删除用户”的行为不一致。
- 若只修前端项目拖拽，不修后端排序作用域，用户一旦存在回收站项目，排序仍可能失败。
- 访视内表单拖拽若不补齐后端批量 reorder 契约，前端只能做到假排序或回弹排序。
- 调浅蓝色主色若未同步检查暗色模式与文本对比度，可能造成可读性下降。
- 导入重名问题在代码层已有处理逻辑；若实际仍报错，真正根因可能在导出文件内容、导入前校验链路或前端错误翻译，存在“误判问题位置”的风险。

## Research Notes

- 本次 research 同时结合了本地代码检索、已有 OpenSpec archive 提案格式参考、以及并行多模型边界探索结果。
- 多模型结论与本地代码核对后，已确认：
  - 用户项目数统计问题是**真实后端约束错误**；
  - 设置与整库导出权限问题是**当前产品边界与用户期望不一致**；
  - 项目导入重名问题与当前代码存在冲突，需在后续阶段进一步定位真实失败路径。
- 本提案只定义约束与成功判据，不代表实现方案已最终定稿。
