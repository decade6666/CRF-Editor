# 提案：在设置中增加普通用户自助修改密码入口

## 目标

在现有“设置”弹窗中，为**普通已登录用户**提供一个自助修改自己密码的入口，避免普通用户只能依赖管理员重置密码。

本次变更只做 OpenSpec 规划，不修改应用源码。

## 已冻结的产品决策

- 按钮文案固定为：`修改密码`
- 入口位置固定为：设置弹窗“当前用户”这一行中，**位于用户名右侧**
- 入口仅普通用户可见；管理员设置弹窗中**不显示**该按钮
- 交互形式固定为：设置弹窗中的子弹窗，不新增独立账号安全页面
- 子弹窗字段固定为：`当前密码`、`新密码`、`确认新密码`
- 提交成功后固定为：**先提示成功，再立即退出当前登录态，并要求用户重新登录**
- production 环境下，自助改密接口固定为**复用现有登录限流契约**

## 交互范围

- 普通用户打开设置弹窗时，可在“当前用户”一行看到用户名与右侧的“修改密码”按钮。
- 点击“修改密码”后，打开独立子弹窗。
- 子弹窗内填写当前密码、新密码、确认新密码。
- 前端在提交前校验“新密码”和“确认新密码”一致；后端只接收 `current_password` 与 `new_password`。
- 后端校验当前密码正确，并复用现有密码策略校验新密码。
- 修改成功后，前端展示成功提示，随后立即清理本地登录态并回到登录界面。

## 后端契约范围

- 新增认证域自助改密接口：`PUT /api/auth/me/password`
- 该接口仅允许已登录的**普通用户**调用；管理员调用返回 `403`
- 请求体固定为：
  - `current_password: string`
  - `new_password: string`
- 成功响应固定为：`204 No Content`
- 修改成功后必须：
  - 更新 `hashed_password`
  - 递增 `auth_version`
  - 使全部旧 JWT 在后续受保护请求中失效
- 当前密码错误、密码策略不满足、与当前密码相同等业务失败固定返回 `400`，不得返回 `401`
- production 下复用现有登录限流，保持 `429 + Retry-After` 契约不变

## 范围边界

- 不包含忘记密码/找回密码流程
- 不包含短信、邮箱、验证码等外部验证机制
- 不改变管理员为他人重置密码的现有流程
- 不为管理员补充新的自助改密入口
- 不引入新的密码复杂度规则；继续复用现有密码策略
- 不引入新的会话表、token 黑名单或多端会话管理机制

## 背景依据

- 设置弹窗位于 `frontend/src/App.vue`
- 当前“当前用户”行是本次按钮落点
- 密码策略、哈希与验密逻辑已存在于 `backend/src/services/auth_service.py`
- JWT 立即失效机制已由 `auth_version` 与 `backend/src/dependencies.py` 实现
- 管理员为他人重置密码已存在于 `backend/src/services/user_admin_service.py`
- production 登录限流已存在于 `backend/src/rate_limit.py`

## 成功判据

1. 普通用户在设置弹窗“当前用户”一行可见用户名右侧的“修改密码”按钮。
2. 管理员在设置弹窗中看不到该按钮。
3. 子弹窗包含 `当前密码 / 新密码 / 确认新密码` 三个字段。
4. 当前密码错误时，请求失败且当前会话不被登出。
5. 新密码不满足现有策略时，请求失败且不修改数据库状态。
6. 新密码与确认新密码不一致时，前端阻止提交。
7. 修改成功后，先提示成功，再立即退出当前登录态并返回登录界面。
8. 修改成功后，旧密码登录失败，新密码登录成功。
9. 修改成功前签发的旧 JWT 在后续受保护请求中返回 `401`。
10. production 下连续失败命中阈值后，自助改密接口返回 `429` 且带 `Retry-After`。
