# Proposal: Word 导出严格对齐参考文档并修复空内容

**Change ID**: strict-word-export-reference-template
**Version**: 1.0.0
**Status**: Research Complete → Ready for Plan (Updated 2026-03-31 with format constraints)
**Date**: 2026-03-30
**Author**: CCG spec-research

---

## 1. 一句话总结

修复 Word 导出出现空内容/空文件的问题，并将导出结果约束为与 `docs/XX项目-eCRF-V1.0-2026XXXX.docx` 在封面、页眉、目录、表单访视分布图、正文表单、图片与页脚骨架上保持严格等价的程序生成文档。

---

## 2. 背景

当前前端导出链路为 `frontend/src/App.vue` 中的两段式下载流程：

1. `POST /api/projects/{id}/export/word/prepare`
2. `GET /api/export/download/{token}`

后端导出入口为 `backend/src/routers/export.py`，由 `ExportService.export_project_to_word()` 负责生成 `.docx`。

研究确认了两个关键现象：

| 现象 | 结论 |
|---|---|
| 用户提供的失败样本 `D:\下载\Chrome Download\通用表单_CRF (1).docx` | 文件大小为 `0` 字节，已是无效 docx，属于真实失败样本 |
| 当前仓库中的参考文档 `docs/XX项目-eCRF-V1.0-2026XXXX.docx` | 为固定骨架文档，实测包含 `41` 段落、`8` 张表、`5` 个 section |
| 当前后端导出实现 | 从空白 `Document()` 程序化生成，不是基于参考文档模板填充 |
| 当前导出结构 | 研究样本下可生成 docx，但结构与参考文档骨架差异显著，不能视为“严格对齐” |

因此，本次需求不是简单样式微调，而是**以参考文档为唯一约束源，重建导出骨架与内容映射规则**。

| H2 | 参考文档中的封面、页眉、目录、表单访视分布图、正文表单、图片与页脚骨架全部属于硬约束，不能只做部分对齐 | 用户确认 |
| H3 | 当项目数据不完整时，仍必须保留参考文档中的空白章节与表格骨架 | 用户确认 |

| H5 | 导出主渲染仍依赖 `ProjectRepository.get_with_full_tree()` 提供的完整项目树，不能绕过现有数据装载链路凭空造数 | 现有代码 |
| H6 | 当前生产依赖仅确认有 `python-docx`，未引入 docxtpl/mailmerge 等模板引擎；研究阶段不得假定存在额外模板库 | 依赖扫描 |
| H7 | 当前项目内部已把”封面信息表 + 访视分布图”视为文档顶层骨架的一部分；`docx_import_service` 默认跳过前 `2` 张表 | 现有代码 |

| # | 约束 | 参考文档值 | 当前导出值 | 修正要求 |

## 5. 已发现的结构事实

### 5.1 当前导出实现事实

- `backend/src/services/export_service.py` 当前从空白 `Document()` 开始依次添加：
  1. 封面页
  2. 页眉页脚
  3. 目录占位
  4. 访视流程图
  5. 表单正文
- 当前正文主要是“一字段一小表”的程序化布局
- 当前实现与参考文档的固定大表骨架差异很大
- 当前导出链路的成功与否，前端只基于接口返回和下载结果提示，**不校验文档是否为有效非空 docx**

### 5.2 参考文档骨架事实

研究样本 `docs/XX项目-eCRF-V1.0-2026XXXX.docx` 具备以下结构特征：

- 固定 section 拓扑：`5` 个 section
- 固定表格骨架：`8` 张表
- 含嵌入图片资源
- 页眉/页脚为模板级结构，不只是普通段落文本
- 正文不是数百张碎片小表，而是少量固定结构表组合

### 5.3 当前风险事实

- 当前导出可在部分项目上生成 docx，但并不能证明满足“严格对齐参考文档”
- 当前失败样本表明至少存在会生成 `0` 字节文件的路径
- 当前测试主要覆盖字段语义，没有整份 docx 的骨架级护栏

## 6. 依赖关系

```text
frontend/src/App.vue 导出按钮
  -> POST /api/projects/{id}/export/word/prepare
    -> backend/src/routers/export.py
      -> ExportService.export_project_to_word()
        -> ProjectRepository.get_with_full_tree()
          -> Project / Visit / VisitForm / Form / FormField / FieldDefinition / CodeList / Unit
        -> field_rendering.py
      -> 临时文件缓存与 token 下载
```

### 跨模块依赖

- `backend/src/routers/export.py`：导出入口、临时文件、token 下载
- `backend/src/services/export_service.py`：导出骨架与文档拼装主逻辑
- `backend/src/repositories/project_repository.py`：完整项目树装载
- `backend/src/services/field_rendering.py`：字段控件与默认值语义
- `backend/src/services/docx_import_service.py`：对导出文档顶层表结构存在隐式耦合（前 2 张表约定）
- `frontend/src/App.vue`：前端导出触发与下载交互

## 7. 风险与缓解

| 风险 | 概率 | 缓解措施 |
|---|---|---|
| 继续在现有“一字段一表”生成器上补丁式调整，最终仍无法收敛到参考文档骨架 | 高 | 在计划阶段把问题视为“骨架重建”，不是样式修补 |
| 修复了下载成功，但仍可能导出 0 字节/无效 docx/正文缺失 | 高 | 成功判据必须包含文件有效性与正文非空验证 |
| 排序仍按名称导致章节顺序与参考文档不一致 | 中 | 计划阶段显式确认业务顺序来源与排序策略 |
| 仅修正文，不处理页眉页脚/图片/section，无法满足“全部保留” | 高 | 将页眉页脚、图片、section 视为同级硬约束 |
| 导出骨架变化破坏当前导入识别假设 | 中 | 计划阶段纳入 `docx_import_service` 的前 2 张表约束评估 |
| 缺少整文档测试，后续回归难以发现结构偏差 | 高 | 补充文档级结构断言与失败样本回归 |

## 8. 用户确认

本次研究阶段已确认以下决策：

- 对齐方式：**程序生成等价**
- 失败样本：**有明确样本**
  - `D:\下载\Chrome Download\通用表单_CRF (1).docx`
- 模板骨架：**全部保留**
  - 封面、页眉、目录、访视分布图、正文表单、图片、页脚均为硬约束
- 数据缺失策略：**保留空白骨架**

### 8.1 格式对齐确认（2026-03-31）

| 项目 | 决策 |
|---|---|
| 封面格式 | **完整封面** - 需实现方案编号/中心编号/筛选号等字段输入框 |
| 访视分布图 | **每访视一列** - 形成访视×表单矩阵，单元格用×标记 |
| 表单名称 | **Heading 1 样式** - 可出现在目录中 |
| 列宽策略 | **自动计算** - 根据内容动态调整 |
| 标签粗体 | **需要粗体** - FormLabel 样式添加粗体 |
| 单元格对齐 | **两端对齐** - 使用 WD_ALIGN_PARAGRAPH.JUSTIFY |
| 横向字段 | **全部字段** - 非 inline 字段也需导出 |
| 横向分节 | **添加分节符** - 保持现有实现 |

## 9. 成功判据（可验证）

- [ ] 导出不再产生 `0` 字节文件或无效 docx 文件
- [ ] 导出文件可被 `python-docx` 正常读取
- [ ] 导出文件包含非空正文，不是仅有空白封面或空目录
- [ ] 导出结果在封面、页眉、目录、访视分布图、正文表单、图片、页脚骨架上与参考文档保持等价
- [ ] 当项目数据缺失时，仍输出参考文档要求的空白章节与表格骨架
- [ ] 文档级验证能断言关键段落、表数量、section 数量、关键表形状与资源存在性
- [ ] 用户提供的失败样本场景在回归后不再出现空内容/空文件

### 9.1 格式对齐判据

- [ ] **封面**：包含方案编号/中心编号/筛选号字段输入框，结构与参考文档一致
- [ ] **访视分布图**：每个访视独立一列，形成访视×表单矩阵
- [ ] **表单名称**：使用 Heading 1 样式，出现在目录中
- [ ] **正文列宽**：根据内容自动计算，接近参考文档比例
- [ ] **标签粗体**：FormLabel 样式字段显示为粗体
- [ ] **单元格对齐**：正文表格使用两端对齐
- [ ] **横向表单**：包含非 inline 字段，前后有分节符
