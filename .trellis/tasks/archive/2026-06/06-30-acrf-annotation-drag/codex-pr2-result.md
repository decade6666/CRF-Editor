三类默认最终采用的是“单常量”，不是分类常量。前端新增了 `field / inline-header / form` 三种 anchor kind，但它们都映射到同一个后端默认偏移 `-120000 EMU`；之所以可以保持单常量，是因为我把 form-domain 标注也改成了相对标题行定位，normal field 继续相对 value/label anchor，inline header 相对 header cell anchor，这样三类在 `Δy=0` 时都能使用同一个默认 `posOffset = -120000 EMU`，同时避免前端再出现原来那种“form 用页坐标、field 用 cell 坐标”的不同源问题。前后端的“同源”做法是：前端在 [acrfAnnotationGeometry.js](/root/github/CRF-Editor/frontend/src/composables/acrfAnnotationGeometry.js:1) 明确镜像后端 `export_service.py` 的几何常量与公式，数值一一对应，当前无需改后端。

EMU↔px 的换算我按 CSS 绝对单位常量写死了：`1in = 914400 EMU = 96 CSS px = 72 pt`，因此 `1cm = 360000 EMU`，`0.01cm = 3600 EMU`，`1px = 9525 EMU`。拖动时前端用 `CSS_PX_PER_01CM = 96 / 2.54 / 100` 把竖直像素位移换算成 `0.01cm` 整数，再做 `[-200, 200]` clamp；预览定位则按 `top = (defaultOffsetEmu + deltaY01cm * 3600) / 360000 cm` 渲染，所以和后端导出的 `posOffset = 默认 + Δy` 保持同一个符号和单位体系。

实现要点上，这次做了四件事。第一，`.wp-acrf-annotation` 现在在 aCRF + editMode 下可竖直拖动，拖动逻辑放在 [useAcrfAnnotationDrag.js](/root/github/CRF-Editor/frontend/src/composables/useAcrfAnnotationDrag.js:1)，拖动过程只做本地乐观更新，松手后才排队 PATCH。第二，PATCH 走 `/api/forms/{id}` 的 `annotation_positions`，字段 key 用 `variable_name`，表单 domain key 用 `_form`，重置操作会直接删除该 key，而不是持久化一个 `0`。第三，保存队列带防抖和合并，多次拖动/重置会合成一次提交；我还特地避开了一个性能坑，乐观更新时不再整对象替换 `selectedForm`，避免误触现有 `watch(selectedForm)` 去反复重拉 `/fields`。第四，`useApi.patch` 新增了可选 `invalidate` 参数，标注保存后会显式失效 `/api/forms/{id}/fields` 和 `/api/projects/{id}/forms` 两类缓存，避免 30 秒 stale。

样式方面，预览标注已经统一成导出红色系：边框 `#C00000`、底色 `#FFF2F2`、文字 `#C00000`。几何也不再写死在 CSS 里，而是通过共享常量生成 CSS 变量，包含字体 `8pt`、高度 `0.7cm`、padding、边框宽度、最大盒宽 `4.6cm`，并且前端的宽度估算公式也镜像了后端的 weighted-char 逻辑。标题区我加了 `.wp-form-title-row`，所以 form-domain 标注现在和字段标注一样是“相对 anchor 定位”，不是相对整页绝对硬编码。
